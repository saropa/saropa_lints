## What's wrong

The rule's `_sensitiveKeywords` set contains the bare word `'token'`. The substring matcher fires on any identifier containing "token" — including `searchTokens`, which is a Drift column / variable holding a denormalized search-index string (a `|`-separated lowercase concatenation of public character-name forms, used to push fuzzy-name lookups to SQLite via `LIKE '%|<needle>|%'`).

These are not auth tokens. They are search-index tokens — the same sense of "token" used in NLP / Lucene / Elasticsearch / SQLite FTS terminology. The values are derived from public character data (Star Trek wiki, Wikidata public-figures), contain no credentials, and have no security implications.

## Why the rule is wrong here

The substring match for `'token'` is too broad. In real-world Dart code, "tokens" appears in many non-credential contexts:

- `searchTokens` — denormalized search-index column (this case)
- `lexerTokens` / `parserTokens` — compiler / parser pipeline
- `wordTokens` / `nGramTokens` — text processing
- `localTokens` — UI gesture token sequences
- `routeTokens` — URL segment lists
- `cspTokens` — Content-Security-Policy directive lists

None are credentials. None should require encryption.

## Suggested fix

Tighten the keyword match. Options in order of preference:

1. **Word-boundary match** with a stop-list of safe prefixes/suffixes:
   - Match `\btoken(s)?\b` where the immediate prefix is NOT one of `search_`, `lex_`, `parser_`, `word_`, `gram_`, `n_gram_`, `route_`, `csp_`, etc.
   - Or invert: require an unambiguous credential-context prefix (`auth`, `api`, `access`, `refresh`, `bearer`, `jwt`, `csrf`, `session`).

2. **CamelCase-aware match**: split identifiers into camelCase / snake_case segments and match `token` only as a *full* segment, not a substring of one. `searchTokens` splits to `['search', 'tokens']` — neither segment is `token` alone (it's `tokens`), but more importantly the leading segment `search` clearly disambiguates the meaning.

3. **Context-aware match**: skip the rule when the variable / column is annotated or named in a way that signals search/index intent — e.g. names ending in `Index`, `Lookup`, `Search`, or containing `_index`, `_lookup`. Cheap and high-precision.

4. **Combine 2 + 3**: full-segment match for the keyword AND prefix-segment exclusion for safe contexts.

The current implementation also flags `'auth'` as a substring — same FP risk on legitimate names like `authorshipMetadata`, `authorPrefix`, `authority` (governmental jurisdictions), etc. The same fix should apply to every entry in `_sensitiveKeywords`.

## Resolution (2026-05-03, rule version v7)

Two complementary changes in `lib/src/rules/security/security_auth_storage_rules.dart`:

1. **Per-keyword regex with safe-context exclusion** — the bare substring entries `'token'` and `'auth'` were removed from `_sensitiveKeywords` and replaced with two dedicated regexes:
   - `_tokenKeywordPattern` matches `tokens?` only when NOT preceded by one of `search`, `lexer`, `parser`, `word`, `gram`, `route`, `csp`, `index`, `fts`, `stop`, `nlp`, `lex`. Real credential prefixes (`auth`, `api`, `access`, `refresh`, `bearer`, `jwt`, `csrf`, `session`, `oauth`, `id`) are deliberately omitted from the exclusion list, so `authToken`, `apiToken`, `accessToken`, etc. still match.
   - `_authKeywordPattern` matches `auth` only when NOT followed by `or` that ISN'T the `oriz` of `authorize`/`authorization`. Truth-table-verified: `author`/`authority`/`authorship`/`authored`/`authoring` no longer match, but `authorize`/`authorization`/`unauthorized`/`authentication`/`authToken`/`oauth` still do.
2. **Same-call search-index context skip** — `_argumentSearchIndexContextPatterns` is checked early and bails out when the args contain unambiguous search-index / parser / NLP field names (`searchTokens`, `searchIndex`, `lookupTokens`, `lookupIndex`, `lexerTokens`, `parserTokens`, `wordTokens`, `(n)gramTokens`, `routeTokens`, `cspTokens`, `indexTokens`, `ftsTokens`, `stopTokens`, `nlpTokens`). This handles the exact case from the bug report — `Companion(searchTokens: Value(tokens))` — where the field name disambiguates the intent of the bare `tokens` value variable.

The rule's user-facing message version is bumped to `{v7}`.

### Downstream

The `tokens` → `searchIndex` local rename in `saropa/contacts` `_migrateToV26` / `_migrateToV27` is no longer required. It is harmless if left in place.

### Accepted residual case

`prefs.setString('cache_key', tokens)` where `tokens` is a bare local variable AND the args contain no other search-index signal will still false-positive — without type/dataflow analysis the rule cannot tell whether the variable holds a search index or an auth token. The downstream workaround is to give the variable a context-bearing name (`searchTokens`, `searchIndex`, `lexerTokens`, etc.) which the same-call context check then suppresses.
