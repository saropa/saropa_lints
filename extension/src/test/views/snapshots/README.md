# Visual snapshot tests — editor-area dashboards

Per [UX guidelines §19.3](../../../../plans/guides/UX_UI_GUIDELINES.md#193-visual-snapshot-tests), every dashboard surface ships with one representative snapshot per default theme so a structural regression breaks the build instead of shipping silently.

## Two layers of coverage

This project takes a deliberately lightweight approach to "visual" testing because:

1. The webview HTML is fully deterministic given a payload — no flaky animation timing, no font fallback drift.
2. Pixel-perfect snapshots (Playwright + Chromium baselines) cost ~30MB of binaries and ~5s per surface; the value is moderate.
3. **Structural** snapshots — capturing the rendered HTML tree, the set of theme tokens referenced, and the computed contrast assumptions — catch the high-value regressions (theme drift, accessibility decay, layout structural breaks) at near-zero cost.

The two layers:

### Layer 1 — HTML structural snapshot (in-tree, fast)

Each `*-html.test.ts` file under `extension/src/test/` already does this implicitly: it asserts on substring presence in the rendered HTML. A more rigorous variant lives in this folder — `<surface>.snapshot.test.ts` files capture a normalized form of the HTML tree and compare against a baseline `<surface>.snapshot.txt` file in the same directory.

**Run:** part of `npm test`. Failures show a unified diff so a contributor sees exactly which substring drifted.

### Layer 2 — Pixel-perfect screenshots (manual, periodic)

Captured manually by opening each dashboard against four theme stubs in a real VS Code window:
- Default Dark+
- Default Light+
- Default High Contrast Dark
- Default High Contrast Light

Screenshots saved as `<surface>-<theme>.png` under this directory. Compared visually during release-readiness review (~quarterly) or when a CSS PR touches a surface.

A future contributor wanting to automate Layer 2 should add `@playwright/test` as a devDependency, write a webview harness that injects theme CSS variables, and emit baselines under this folder. The structure above is ready for that drop-in.

## Token coverage matrix

Generated by `scripts/build-token-matrix.ts` (run `npx ts-node scripts/build-token-matrix.ts`). Walks every `*-styles.ts` file, extracts `var(--vscode-*)` references, and emits `extension/src/test/views/snapshots/token-matrix.md` — a markdown table of tokens × surfaces. Re-run after every CSS change; CI flags the diff.
